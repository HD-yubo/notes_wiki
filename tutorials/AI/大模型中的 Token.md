# 大模型中的 Token：从字符到数字的完整旅程


你每天都在和大语言模型对话，但你输入的文字在模型内部经历了什么？模型并不"阅读"你的文字——它看到的是一串数字。理解 Token 这个概念，是理解大模型工作原理的第一步，也是理解成本、上下文限制和多语言差异的关键。

---

## 第1章 什么是 Token——大模型眼中的文字

### 1.1 模型不读文字，模型读数字

想象你打开一个翻译软件，把一段中文粘贴进去。你看到的是文字输入和文字输出，但在翻译软件的内部，文字已经被拆成了更小的片段，经过处理后重新组合。

大语言模型做的本质上是同一件事，只是粒度不同。当你输入"今天天气真好"时，模型看到的不是这六个汉字本身，而是一串编号：

```
输入文本:  今天天气真好
Token IDs: [2765, 10136, 3221, 2523]
```

每个编号对应词表中的一个 Token。模型的所有运算——理解、推理、生成——都是在这串数字上进行的。

### 1.2 什么是 Token

Token 是大语言模型处理文本的最小单位。它不是一个固定的"字"，也不是一个固定的"词"，而是一个由分词算法（Tokenizer）根据统计规律切分出的文本片段。

一个 Token 可以是：
- 一个完整的英文单词，如 `hello`
- 一个英文单词的一部分，如 `un` + `believe` + `able`
- 一个中文词或字，如 `今天` 或 `天`
- 一个标点符号，如 `.`
- 一个空格或换行符

### 1.3 为什么不用字或词作为单位

最直觉的做法是按字切分（每个中文字一个 token）或按词切分（每个英文词一个 token），但两种方式都有严重问题：

按字切分：英文中 "unbelievable" 会变成 13 个 token，序列太长，效率低下。
按词切分：遇到新词（如互联网新造词 "rizz"）就无法处理，模型词表覆盖不了所有可能的词。

子词分词（Subword Tokenization）在两者之间找到平衡：高频词保持完整，低频词和新词被拆成更小的片段。这让词表大小可控，序列长度合理，且能处理任何输入。

### 1.4 关键要点

- Token 是模型处理文本的基本单位，不是字也不是词
- 模型通过 Token ID（整数编号）而非原始文字来处理信息
- 子词分词平衡了词表大小和序列效率
- 不同模型的 Tokenizer 切分方式不同，同一段文字的 Token 数可能不同

## 第2章 分词算法：BPE、WordPiece 与 SentencePiece

### 2.1 三种主流方法

目前大语言模型使用的主要分词算法有三种。它们的共同目标是：用有限的词表覆盖无限的文本。

| 算法                       | 合并方向     | 代表模型                | 核心思想                     |
| ------------------------ | -------- | ------------------- | ------------------------ |
| BPE (Byte Pair Encoding) | 从小到大逐步合并 | GPT 系列、Llama        | 统计最高频的相邻符号对，反复合并         |
| WordPiece                | 从小到大逐步合并 | BERT、早期 Transformer | 基于似然增益选择合并对              |
| Unigram / SentencePiece  | 从大到小逐步删减 | T5、LLaMA、Gemini     | 从大词表开始，逐步移除信息损失最小的 token |

### 2.2 BPE：从数据压缩到语言模型

BPE 最初是一种数据压缩算法。它的核心思想非常简单：

1. 把文本拆成最小单元（字符或字节）
2. 统计所有相邻符号对出现的频率
3. 把出现最多的符号对合并成一个新的 token
4. 重复步骤 2-3，直到达到目标词表大小

举个例子。假设语料中 `t` 和 `h` 相邻出现了 100 次，是所有相邻对中最高的，那就把 `th` 合并成一个新 token。下一轮，`th` 和 `e` 相邻出现了 80 次，合并成 `the`。如此迭代。

BPE 训练的最终产物不是词表本身，而是一组合并规则（merges）。分词时按这些规则的顺序依次应用。

### 2.3 WordPiece 与 Unigram

WordPiece 的思路和 BPE 类似，也是从小到大合并，但它选择合并对的标准不是纯频率，而是合并后对整体词表似然的增益。这在理论上更优雅，但实际效果与 BPE 相当。

Unigram 的方向相反：它从一个很大的词表开始，逐步移除那些移除后对整体似然损失最小的 token。最终保留下来的 token 构成词表。

### 2.4 SentencePiece：不是算法，是工具

SentencePiece 是一个开源库，不是一种分词算法。它的特别之处在于：可以直接在原始文本上操作，不需要预先按空格切分单词。这对于中文、日文等没有明确空格分隔的语言特别重要。

### 2.5 关键要点

- BPE 是最主流的方法，GPT 系列和 Llama 都在使用
- BPE 的产物是合并规则列表，不只是词表
- SentencePiece 是工具库，支持 BPE 和 Unigram 两种算法
- 不同模型的 Tokenizer 切分方式不同，同一个词在不同模型中可能是不同数量的 token

![图 2.1：三种分词算法的核心对比](assets/llm_token/chapter-02.png)
*图 2.1：三种分词算法的核心对比——BPE 和 WordPiece 从小合并，Unigram 从大删减。*

> **常见陷阱**：不要说"所有大模型都用 BPE"。BERT 用 WordPiece，T5 用 SentencePiece + Unigram。算法选择取决于模型设计者。

## 第3章 一个完整的分词实例

### 3.1 用 BPE 走一遍完整流程

假设我们有一个小型语料库，包含以下句子：

```
"low low low low low"
"new new new new"
"newer newer newer"
"widest widest"
"newest newest"
```

**第一步：预分词**

按空格切分，并统计每个词的频率：

```
l o w </w>        ×5
n e w </w>        ×4
n e w e r </w>    ×3
w i d e s t </w>  ×1
n e w e s t </w>  ×2
```

每个词被拆成字符序列，`</w>` 标记词尾。

**第二步：统计相邻对频率**

```
(e, r)   = 5
(n, e)   = 9
(l, o)   = 5
(o, w)   = 5
(w, </w>) = 10
(s, t)   = 3
(e, s)   = 3
...
```

最高频的是 `(e, r)` = 5？不对，`(n, e)` = 9 更高。但更重要的是 `(w, </w>)` = 10。让我们重新统计：

- `n` `e` 在 "new"×4 + "newer"×3 + "newest"×2 = 9 次相邻
- `w` `</w>` 在 "low"×5 + "new"×4 = 9 次相邻

**第三步：第一轮合并**

最高频对是 `(n, e)` = 9。合并后得到新 token `ne`。

**第四步：第二轮合并**

现在语料中出现 `ne` 的地方：`ne w`、`ne w e r`、`ne w e s t`。最高频对变为 `(ne, w)` = 9。合并为 `new`。

**第五步：继续迭代**

随着合并进行，越来越大的片段被识别为 token。最终 `newest` 可能成为单个 token，而罕见的 `widest` 可能保持为 `w i d e s t` 多个 token。

### 3.2 中文分词示例

中文没有空格，Tokenizer 通常先做字级切分，再用 BPE 合并高频组合。

```
输入: "我喜欢编程"
初始: [我, 喜, 欢, 编, 程]
经过 BPE 合并后: [我, 喜欢, 编程]
```

但这取决于训练语料。如果 "编程" 在语料中出现频率不高，可能被切分为 [编, 程]。这就是为什么不同模型对同一段中文的 token 数可能不同。

### 3.3 关键要点

- 分词过程从最小单元开始，通过统计合并逐步构建 token
- 训练语料决定哪些片段被合并为 token
- 中文分词依赖字级起点 + BPE 合并，结果因模型而异
- token 数量不能简单按字数估算

![图 3.1：BPE 的逐步合并过程](assets/llm_token/chapter-03.png)
*图 3.1：BPE 的逐步合并过程——从字符开始，每轮合并最高频相邻对。*

## 第4章 从 Token 到向量：Embedding 的角色

### 4.1 模型怎么"理解"一个 token

分词完成后，模型得到的是一串整数 ID，例如 `[2765, 10136, 3221]`。但整数本身没有语义。模型需要的是一个能携带含义的表示——这就是 Embedding（嵌入向量）。

Embedding 的过程可以这样理解：

```
Token "喜欢" → Token ID: 10136 → Embedding 向量: [0.23, -0.47, 0.81, ..., 0.12]
```

这个向量通常有几百到几千个维度（具体取决于模型）。每个维度不是独立可读的含义，但整个向量编码了这个 token 的语义信息。

### 4.2 Embedding 矩阵

模型内部维护一个巨大的 Embedding 矩阵，行数等于词表大小，列数等于向量维度。分词后的 Token ID 就是这个矩阵的行索引——一次查表操作就能得到向量。

```
Embedding 矩阵 (词表大小 × 向量维度)
┌─────────────────────────────────┐
│  ID 0   → [0.12, -0.34, ...]   │
│  ID 1   → [0.56, 0.78, ...]    │
│  ...                            │
│  ID 10136 → [0.23, -0.47, ...] │  ← "喜欢" 的向量
│  ...                            │
│  ID 50000 → [-0.11, 0.65, ...] │
└─────────────────────────────────┘
```

### 4.3 向量之后发生了什么

得到向量后，Transformer 架构开始工作：

1. **Self-Attention（自注意力）**：每个 token 的向量与其他所有 token 的向量交互，让模型理解词与词之间的关系。比如"他"和"张三"之间的指代关系就在这一步建立。
2. **前馈网络（MLP）**：进一步变换每个 token 的表示，提取更复杂的模式。
3. **多层堆叠**：上述过程重复几十层，每层都在不同的抽象层次上理解文本。
4. **输出预测**：最后一层的输出用于预测下一个 token 的概率分布。

### 4.4 关键要点

- Token ID 是 Embedding 矩阵的索引，一次查表得到向量
- Embedding 向量的维度取决于具体模型（通常几百到几千维）
- 向量编码语义信息，让相似含义的 token 在向量空间中距离更近
- Self-Attention 让 token 之间可以互相"看到"，建立上下文关系

![图 4.1：Token 在模型中的处理层级](assets/llm_token/chapter-04.png)
*图 4.1：Token 在模型中的处理层级——从一个整数编号逐步变成语义丰富的向量表示。*

> **检查点**：如果你删除 Embedding 矩阵的某些行（改变词表），模型就无法处理那些 token。这就是为什么 Tokenizer 的设计如此重要——它决定了模型"认识"哪些文本片段。

## 第5章 Token 与成本：你的钱花在哪了

### 5.1 为什么 API 按 token 收费

大语言模型 API 的计费单位是 token。这不只是商业策略——token 数量直接决定了模型需要的计算量。每一次推理（无论是理解输入还是生成输出），模型都在对每个 token 做矩阵运算。token 越多，计算越多，资源消耗越大。

### 5.2 输入 token 与输出 token

API 通常将计费分为两部分：

| 类型 | 说明 | 相对成本 |
| --- | --- | --- |
| 输入 token（Input / Prompt） | 你发送给模型的内容 | 较低 |
| 输出 token（Output / Completion） | 模型生成的内容 | 较高，通常 2-4 倍 |

输出 token 更贵的原因：生成每个输出 token 都需要一次完整的前向传播，且通常需要更复杂的计算（如采样、温度调整）。

### 5.3 各模型定价参考

以下是截至 2025-2026 年的大致定价（每百万 token），具体以官方定价页为准：

| 模型 | 上下文窗口 | 输入价格 | 输出价格 |
| --- | --- | --- | --- |
| GPT-4 Turbo | 128K | ~$10 | ~$30 |
| GPT-4.1 | 1M | ~$2 | ~$8 |
| Claude Sonnet | 200K | ~$3 | ~$15 |
| Gemini 2.5 Flash | 1M | ~$0.30 | ~$1.25 |

### 5.4 一个粗略的估算方法

如果你在使用中文，一个粗略的估算规则是：1 个中文字 ≈ 1-1.5 个 token。英文中，1 个常见英文单词 ≈ 1 个 token，但短词（a, the, is）可能被合并到相邻词中。

实际 token 数取决于具体模型的 Tokenizer。最准确的方式是使用模型提供的 token 计数工具或 API 端点。

### 5.5 关键要点

- API 按 token 收费，输出 token 通常比输入贵 2-4 倍
- 不同模型的定价和上下文窗口差异很大
- token 数量不能简单按字数估算，最准确的方式是使用实际计数
- 节省成本的方法：精简输入、控制输出长度、选择合适的模型

## 第6章 Token 与上下文窗口：记忆的边界

### 6.1 什么是上下文窗口

上下文窗口（Context Window）是模型一次能处理的最大 token 数量。它包括输入 token 和输出 token 的总和。

可以把它想象成一张桌子的面积：你放入的文档越多（输入 token），留给模型书写答案的空间（输出 token）就越少。

### 6.2 各模型的上下文容量

| 模型 | 上下文窗口 |
| --- | --- |
| GPT-4 Turbo | 128K tokens |
| Claude 3.5 Sonnet | 200K tokens |
| Gemini 2.5 Pro | 2M tokens |

### 6.3 上下文窗口的限制意味着什么

- **对话历史有上限**：当对话足够长时，早期的对话内容会丢失或被压缩
- **长文档处理有限**：超过上下文窗口的文档需要分段处理
- **system prompt 占用空间**：系统提示词也计入 token，过长的系统提示会压缩可用空间
- **成本随上下文增长**：发送更长的上下文意味着更高的输入 token 费用

### 6.4 上下文窗口之外

即使文本在上下文窗口内，模型对不同位置的 token 的"关注度"也不同。研究表明：

- 模型对输入的开头和结尾关注度较高（"中间丢失"现象）
- 超长上下文中，远距离的关联可能被忽略
- 上下文窗口大小 ≠ 有效记忆深度

### 6.5 关键要点

- 上下文窗口是模型一次能处理的总 token 数（输入+输出）
- 不同模型的窗口从 8K 到 2M tokens 不等
- 长上下文不等于有效记忆，模型可能忽略中间部分的信息
- 选择合适的上下文长度需要平衡能力、成本和实际需求

![图 6.1：主流模型的上下文窗口对比](assets/llm_token/chapter-06.png)

## 第7章 多语言视角：中文 Token 的特殊性

### 7.1 中文的 Token 效率问题

直觉上，中文每个字信息密度高，应该比英文更"省 token"。但现实往往相反。

原因在于 Tokenizer 的训练数据。大多数主流模型（GPT、Claude）的训练语料以英文为主，英文的高频词汇和短语被完整合并为 token，而中文字符在词表中的覆盖相对不足。

### 7.2 实际测试数据

根据多个独立测试的汇总：

- 在 GPT 和 Claude 的 Tokenizer 上，中文通常比等义英文消耗 **1.1 倍到 2.9 倍** 的 token
- 在 Qwen 和 DeepSeek 等中文优化模型上，中文的 token 效率与英文相当，甚至更好
- 简体中文通常比繁体中文效率略高

### 7.3 为什么不同模型差异这么大

这取决于 Tokenizer 训练时使用的语料组成：

- 训练语料中某种语言占比越高，该语言的 token 效率通常越好
- 中文优化模型（如 Qwen、DeepSeek）在训练时使用了更多中文语料，所以中文分词更高效
- Tokenizer 词表中分配给中文的 token 数量也影响效率

### 7.4 实用建议

- 如果成本敏感，优先使用对目标语言优化的模型
- 不要为了"省 token"而把所有提示词翻译成英文——这可能增加理解偏差
- 对于频繁使用的提示词，可以用 token 计数器实际测量

### 7.5 关键要点

- 中文在很多主流模型上比英文消耗更多 token
- 这是训练语料和词表设计的结果，不是语言本身的差异
- 中文优化模型（Qwen、DeepSeek）的中文 token 效率更好
- 节省成本的最佳方式是选择适合的模型，而不是翻译语言

![图 7.1：不同模型上中文相对于英文的 token 效率](assets/llm_token/chapter-07.png)
*图 7.1：不同模型上中文相对于英文的 token 效率——选择对目标语言优化的模型是关键。*

## 第8章 常见误区与实用建议

### 8.1 误区一：Token = 字或词

事实：Token 是分词算法的产物，大小不固定。一个英文单词可能是一个 token，也可能是多个。一个中文词可能是一个 token，也可能被拆成更小的片段。

### 8.2 误区二：所有模型的 Tokenizer 相同

事实：不同模型使用不同的 Tokenizer。GPT-4 用的是 tiktoken（BPE 变体），Claude 用的是自己的 BPE 实现，Qwen 用的是基于 SentencePiece 的分词器。同一段文字在不同模型中的 token 数可能相差 30% 以上。

### 8.3 误区三：英文一定比中文省 token

事实：在 GPT 和 Claude 上确实如此，但在 Qwen、DeepSeek 等中文优化模型上，中文可能更省。选择适合语言的模型比翻译语言更有效。

### 8.4 误区四：上下文窗口越大越好

事实：大上下文窗口提供更多灵活性，但也有代价：成本更高、中间信息可能被忽略、推理速度可能下降。选择合适的窗口大小比选择最大的更重要。

### 8.5 误区五：token 数量可以精确估算

事实：没有简单的换算公式。最准确的方式是使用模型的 token 计数器或 API。

### 8.6 实用检查清单

| 场景 | 建议 |
| --- | --- |
| 日常对话 | 不需要特别关注 token |
| API 调用 | 使用 token 计数器预估成本 |
| 长文档处理 | 分段处理，注意上下文限制 |
| 多语言任务 | 选择对目标语言优化的模型 |
| 成本优化 | 精简输入、控制输出长度、选择合适模型 |
| 提示词工程 | 使用 token 计数器验证提示词效率 |

### 8.7 关键要点

- Token 的大小和数量取决于具体模型的 Tokenizer
- 不同模型的 Tokenizer 不同，不要假设通用规则
- 选择适合的模型比调整语言或格式更有效
- 最准确的信息来源是实际测量，不是估算公式
